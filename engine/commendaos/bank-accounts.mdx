## Overview

Use bank accounts to manage financial institution information for your business entities. Bank accounts store account details, routing information, and country-specific credentials for corporations and other legal entities.

Bank accounts support a global, content-based model where the required fields adapt to the country where the account is held. A fields configuration endpoint lets you retrieve the exact field requirements for each country at runtime, so your integration can dynamically render the correct form.

## Prerequisites

Before adding a bank account, ensure you have:

- A business entity created in CommendaOS
- The business entity ID
- Bank account details (account number, routing number or equivalent, bank name)
- Account type information (`Checkings`, `Savings`, etc.)
- The country where the bank account is held (ISO 3166-1 alpha-2 code)

## Retrieve country-specific field requirements

Before creating a bank account, call the fields configuration endpoint to discover which credential fields are required or optional for the target country.

```http
GET /bank-accounts/fields-config
```

The response contains:

- **`field_definitions`** â€” all supported bank account fields, their types, labels, and validation rules.
- **`countries`** â€” a mapping of ISO country codes to the required and optional credential fields for that country, along with any country-specific validation overrides.

Use this data to render the correct form fields based on the selected country. Fields marked with `required_globally: true` in the definitions (such as `accountHolderName`, `country`, `currency`, and `bankName`) apply to every country. All other fields are country-specific credentials.

**Example response (partial)**

```json
{
  "field_definitions": {
    "accountHolderName": {
      "label": "Account Holder Name",
      "type": "string",
      "required_globally": true,
      "description": "Full legal name of the account holder"
    },
    "routingNumber": {
      "label": "Routing Number (ABA)",
      "type": "string",
      "validation": { "pattern": "^[0-9]{9}$", "length": 9 },
      "description": "US ABA routing transit number"
    },
    "ifscCode": {
      "label": "IFSC Code",
      "type": "string",
      "validation": { "pattern": "^[A-Z]{4}0[A-Z0-9]{6}$", "length": 11 },
      "description": "Indian Financial System Code"
    }
  },
  "countries": {
    "US": {
      "name": "United States",
      "primary_currency": "USD",
      "required_fields": ["routingNumber", "accountNumber"],
      "optional_fields": ["accountType"]
    },
    "IN": {
      "name": "India",
      "primary_currency": "INR",
      "required_fields": ["ifscCode", "accountNumber"],
      "optional_fields": []
    },
    "GB": {
      "name": "United Kingdom",
      "primary_currency": "GBP",
      "required_fields": ["sortCode", "accountNumber"],
      "optional_fields": []
    }
  }
}
```

<Tip>
Cache the fields configuration on the client side and refresh it periodically. The configuration rarely changes but may be updated when new countries are added.
</Tip>

## Add a bank account

To add a new bank account to a corporation:

### Step 1: Gather required information

Collect the following base fields (required for all countries):

- **Bank name** â€” full legal name of the financial institution
- **Account holder name** â€” legal name on the account (should match business entity)
- **Country** â€” ISO 3166-1 alpha-2 code (e.g., `US`, `IN`, `GB`)
- **Currency** â€” ISO 4217 currency code (e.g., `USD`, `INR`, `GBP`)

Then collect the country-specific credential fields. Use the [fields configuration endpoint](#retrieve-country-specific-field-requirements) to determine which fields are required for the target country.

### Step 2: Create the bank account

```http
POST /business-entity/:businessEntityId/bank-accounts
```

**Path parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `businessEntityId` | integer | Unique identifier for the business entity |

**Request body**

Pass country-specific routing and account details inside the `credentials` object. The example below creates a US checking account:

```json
{
  "bankName": "First National Bank",
  "accountHolderName": "Acme Corporation",
  "country": "US",
  "currency": "USD",
  "credentials": {
    "accountNumber": "123456789",
    "routingNumber": "021000021",
    "accountType": "Checkings"
  },
  "isPrimary": true
}
```

<Info>
Account and routing details are canonical inside the `credentials` object. Top-level `accountNumber`, `routingNumber`, and `swiftCode` fields are still accepted for backward compatibility and are automatically mirrored into `credentials.accountNumber`, `credentials.routingNumber`, and `credentials.swiftBic`.
</Info>

For an Indian bank account, the `credentials` object uses different fields:

```json
{
  "bankName": "State Bank of India",
  "accountHolderName": "Acme India Pvt Ltd",
  "country": "IN",
  "currency": "INR",
  "credentials": {
    "accountNumber": "123456789012",
    "ifscCode": "SBIN0001234"
  }
}
```

<Info>
When `credentials.accountType` is set, its value is used directly for both `bankAccountType` and `subType` on the stored record. Supported values are `Checkings` and `Savings`. If you omit `credentials.accountType`, it is derived from the top-level `bankAccountType` and `subType` fields when provided.
</Info>

**Response**

```json
{
  "id": "ba_abc123",
  "businessEntityId": 123,
  "bankName": "First National Bank",
  "accountHolderName": "Acme Corporation",
  "country": "US",
  "currency": "USD",
  "accountNumber": "****6789",
  "credentials": {
    "accountNumber": "123456789",
    "routingNumber": "021000021",
    "accountType": "Checkings"
  },
  "bankAccountType": "Checkings",
  "subType": "Checkings",
  "isPrimary": true,
  "createdAt": "2024-01-15T10:30:00Z"
}
```

### Step 3: Verify the account

After creating the bank account, verify the details are correct:

```http
GET /business-entity/:businessEntityId/bank-accounts/:bankAccountId
```

**Path parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `businessEntityId` | integer | Business entity ID |
| `bankAccountId` | string | Bank account ID |

## Update a bank account

To update existing bank account information:

### Step 1: Retrieve current details

First, get the current bank account information:

```http
GET /business-entity/:businessEntityId/bank-accounts/:bankAccountId
```

### Step 2: Update the account

```http
PATCH /business-entity/:businessEntityId/bank-accounts/:bankAccountId
```

**Path parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `businessEntityId` | integer | Business entity ID |
| `bankAccountId` | string | Bank account ID |

**Request body**

Include only the fields you want to update. All fields are optional.

| Field | Type | Description |
|-------|------|-------------|
| `bankName` | string | Full legal name of the financial institution |
| `accountHolderName` | string | Legal name on the account |
| `country` | string | ISO 3166-1 alpha-2 country code where the account is held |
| `currency` | string | ISO 4217 currency code (e.g., `USD`, `EUR`, `INR`) |
| `bankAddress` | string | Address of the bank |
| `taxRegistrationId` | string | Linked tax registration identifier |
| `isPrimary` | boolean | Whether this account is the primary account for the entity |
| `credentials` | object | Country-specific routing and identification fields |

```json
{
  "accountHolderName": "Acme GmbH",
  "country": "DE",
  "currency": "EUR",
  "credentials": {
    "iban": "DE89370400440532013000"
  }
}
```

<Info>
Credential updates are merged with existing values. Sending `{ "credentials": { "iban": "..." } }` adds or overwrites the `iban` field without removing other credential fields already stored on the account.
</Info>

<Info>
Credential keys that are not part of the country's field configuration are silently dropped before validation. This lets you safely update accounts that were created when the legacy schema (for example, a top-level `swiftCode`) was still in use, without having to clean up the existing record first.
</Info>

**Response**

Returns the updated bank account object with `200 OK` status.

## Remove a bank account

To delete a bank account from a corporation:

```http
DELETE /business-entity/:businessEntityId/bank-accounts/:bankAccountId
```

**Path parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `businessEntityId` | integer | Business entity ID |
| `bankAccountId` | string | Bank account ID |

**Response**

Returns `204 No Content` on success.

<Warning>
You cannot delete a bank account if managed filings is enabled for one or more registrations linked to the corporation. Disable managed filings first.
</Warning>

## Credentials reference

The `credentials` object holds country-specific bank routing and identification fields. Refer to the [fields configuration endpoint](#retrieve-country-specific-field-requirements) for the full list, or see common examples below.

| Field | Countries | Description |
|-------|-----------|-------------|
| `accountNumber` | Most | Local bank account number |
| `routingNumber` | US | ABA routing transit number (9 digits) |
| `accountType` | US | `Checkings` or `Savings` |
| `ifscCode` | IN | Indian Financial System Code (11 characters) |
| `sortCode` | GB, IE | Bank sort code (6 digits) |
| `iban` | EU, SEPA countries | International Bank Account Number |
| `swiftBic` | Global | SWIFT/BIC code (8 or 11 characters) |
| `bsb` | AU | Bank/State/Branch code (6 digits) |
| `transitNumber` | CA | Canadian transit number (5 digits) |
| `institutionNumber` | CA | Canadian institution number (3 digits) |
| `clabe` | MX | Mexican standardized bank code (18 digits) |
| `bankCode` | Various | Institution identifier (format varies by country) |
| `branchCode` | Various | Branch identifier (format varies by country) |

## Best practices

- **Use the fields config** â€” always call the fields configuration endpoint to determine required fields for a country instead of hardcoding field lists
- **Set a primary account** â€” mark one account as primary for default transactions
- **Verify account numbers** â€” double-check account and routing numbers before saving
- **Pass the country** â€” always include `country` when creating a bank account so server-side validation can enforce the correct credential rules
- **Update regularly** â€” keep bank account information current when accounts change
- **Secure storage** â€” account numbers are automatically masked in responses for security
- **Currency matching** â€” ensure the currency matches the country's primary currency or your business operations
- **Account holder names** â€” use the exact legal name registered with the bank
